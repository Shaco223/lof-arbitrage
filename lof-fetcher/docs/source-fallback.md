# 真实数据源失败兜底策略

## 目标

免费公开接口存在限速、字段变更、临时不可用风险。本策略用于指导后续 M1.1/M1.2/M1.3 填实时，在不突破 PRD §6 契约与 uniCloud 免费配额的前提下保证可观测、可降级、可恢复。

## 数据源分层

| 数据类别 | 主源 | 备源 | 失败影响 |
| --- | --- | --- | --- |
| LOF 二级市场价格 | 新浪行情 | 东方财富行情 | 无价格则跳过该基金本分钟写入 |
| 重仓股价格 | 新浪行情 | 东方财富行情 | 覆盖率可能降级为 `degraded` |
| 指数行情 | 东方财富/新浪指数 | 集思录或上一分钟缓存 | 指数型 LOF 可能降级为 `degraded` 或 `stale` |
| 基金元数据 | 东方财富 `pingzhongdata` | 天天基金/人工 CSV | 只影响 M1.2 验证与校准，不盘中频繁拉取 |
| 官方 NAV | 天天基金/东方财富净值 | 基金公告人工补录 | 影响 18:00 校准，不影响盘中实时 ingest |

## HTTP 客户端规则

- 使用 `httpx`，统一 `timeout=10s`，`trust_env=False`，避免本机代理污染。
- 每个请求最多重试 3 次：立即重试、延迟 5 秒、延迟 15 秒。
- 单个数据源失败不阻断整个批次；按基金维度记录失败原因。
- 任何真实 token、cookie、代理配置不得入库。

## 降级状态

| 状态 | 触发条件 | 写入策略 |
| --- | --- | --- |
| `ok` | 主源或备源成功，覆盖率 ≥ 90% | 正常写入 `lof-realtime` |
| `degraded` | 部分重仓/指数失败，覆盖率 70%~90% | 写入，但前端展示黄色提示 |
| `stale` | 连续两分钟关键源失败，或覆盖率 < 70% | 默认跳过该基金本分钟写入；如为调试环境可写入但不得触发告警 |

## 批次策略

1. 每分钟构造 30 只 LOF 的候选列表。
2. 对每只基金独立拉取价格与估算 IOPV。
3. `stale` 条目默认剔除，剩余条目组成一个 `lof-ingest` 请求。
4. 若本分钟有效条目为 0，不调用 `lof-ingest`，只写本地日志。
5. 同一分钟最多 1 次 ingest 请求，避免突破 AC-S1 配额。

## 缓存策略

- 元数据、benchmark mapping、前十大持仓：盘前加载到内存，盘中不重复拉取。
- 行情缓存：只允许使用上一分钟缓存做备援；超过 2 分钟视为 `stale`。
- 官方 NAV：只在每日校准任务使用，不参与盘中实时净值估算的字段契约变更。

## 告警策略

- 只有 `source_quality = ok` 或 `degraded` 且处于交易时段时允许进入阈值判断。
- `stale` 不触发告警。
- 同基金 30 分钟冷却；冷却拦截仍写 `alert_log.status = cooldown_blocked`。
- 推送通道失败时写 `alert_log.status = failed`，不重试超过 3 次。

## 日志与排障

日志至少包含：

- `ts`：分钟时间戳。
- `code`：LOF 代码。
- `source`：数据源名称。
- `attempt`：第几次尝试。
- `error_type`：网络错误、HTTP 状态、解析错误、字段缺失。
- `source_quality`：最终降级状态。

本地日志只保留最近 7 天，避免占用磁盘。

## 触发回推 dev-001 的情形

- 数据源字段结构变化导致 PRD §6 输出字段无法满足。
- watchlist 条目确认已转型、清盘或不再满足规模/成交额要求。
- 需要新增或删除接口字段。
- 免费源不可用需要切换到登录态、cookie 或付费数据源。

以上情形均先形成报告，不直接修改 PRD 或 watchlist CSV。
