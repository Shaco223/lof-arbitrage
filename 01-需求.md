# 01-需求 / PRD 1.3（LOF 实时溢价监控）

> 版本：PRD 1.3｜状态：草稿（申赎状态转正：`subscribe_status`/`redeem_status` 由占位 unknown 升真实值并统一枚举；CCR 新增申购额度字段 `subscribe_limit_amount`/`subscribe_limit_period`；基于 dev-004 探测 POC 2839fad；接口名/list/detail 不变）｜责任人：产品经理(dev-002)｜锁定日期：2026-06-24

## 快速摘要

- 背景：场内 LOF 现有炒股软件展示的溢价基于 T-1 净值，盘中失真 1~2%，吃掉全部可交易信号。
- 核心目标：自建"档 3 + 业绩比较基准补全（工业级）"实时溢价计算与监控系统，目标精度 ≥ 90%。
- 一期范围：仅国内 LOF（QDII 二期）／30 只热门自选池／1 分钟刷新／网页 Dashboard + 阈值推送／30 天分位回看／仅二级市场溢价路径。
- 架构：本机 Python 拉取器（`lof-fetcher/`） + uniCloud 免费云空间（`uniCloud-aliyun/`） + uniapp 前端（`frontend/`，H5 → 小程序）。
- 配套交付：`assets/lof-watchlist-v1.csv` / `assets/lof-watchlist-v2.csv`（v2 为当前推荐池）、`assets/benchmark-mapping-v1.csv` / `assets/benchmark-mapping-v2.csv`、`assets/benchmark-validate-v2.md`（v2 权重和自检报告）。
- 远程仓库：https://github.com/Shaco223/lof-arbitrage （主干 `main`，本次修订分支 `feat/prd-v1.3-subscribe-status`）。

## 1. 背景与目标

### 1.1 痛点
- 软件溢价 = (现价 − 昨日净值) / 昨日净值，T-1 数据导致盘中失真。
- LOF 可交易套利空间多在 1~3%，与误差同量级，信号几乎不可用。

### 1.2 目标
1. 盘中每分钟输出 30 只 LOF 真实实时溢价率，整体覆盖率 ≥ 90%。
2. 提供 30 天折溢价历史分位，支持判断当前溢价相对位置。
3. 阈值触发时主动推送告警，避免人工盯盘。
4. 前端使用 uniapp 框架，一期 H5，二期可平滑迁移到微信小程序。

## 2. 一期范围与边界

### 2.1 本期包含
- 标的：国内 LOF（普通股票/债券/混合型 LOF），30 只自选池。
- 算法：档 3 + 业绩比较基准补全（详见 §M2）。
- 刷新：盘中（09:30–11:30 / 13:00–15:00）每 1 分钟；盘后 18:00 用官方净值校准。
- 数据源：东方财富 / 新浪 / 天天基金 / 集思录 公开免费接口。
- 历史：30 天折溢价分布 + 当前分位百分比。
- 套利路径：仅二级市场买卖溢价。
- 告警：溢价 > 5% 或折价 < −2%；同基金 30 分钟冷却；推送 Server酱或邮件二选一。
- 前端：uniapp H5。

### 2.2 本期不包含
- QDII LOF、申赎套利、全市场扫描、付费数据源、自动下单、多用户体系。
- 套利结果自动计算（二期）。
- 新增份额日变化 `shares_change_daily`：免费源覆盖差，延迟 T+1 不适合盘中决策；后续若 dev-004 找到稳定免费源再走 CCR 升 PRD 1.3。
- 不动 list/detail/history/ingest 四个接口名（PRD 1.2 仅做字段扩展，旧字段全部保留）。
- 不放行 AC-S1：仍为 hard pending，等待 3 个真实交易日 uniCloud 配额证据。
- 不打线上 uniCloud：PRD 1.2 验收阶段仍走本地 API `http://127.0.0.1:8787`，不消耗 RU/WU。

## §M2 估算覆盖率公式（**M1 必改修订点**）

### M2-A 覆盖率定义按基金类型分流

设：
- `W_top10` = 前十大重仓股权重之和（来源：季报披露）
- `W_benchmark_assigned` = 已赋指数代码的基准组成项权重之和（不含 CASH）
- `W_cash` = 现金/活期组件权重之和（`index_code = CASH`）

#### 公式 1：指数型 LOF（`type = index`）
> 指数型基金为全样本/抽样复制基准指数，盘中实时净值 ≈ 基准指数实时变动。

`
coverage_index = W_benchmark_assigned + W_cash      # 通常 ≈ 1.00
`

适用条件：`lof_meta.type == 'index'`。
说明：不再使用前十大重仓权重和；前十大重仓股仅用于详情页展示，不进入覆盖率计算。

#### 公式 2：行业型 / 主动型 LOF（`type = industry` / `type = active`）
> 主动管理或行业增强，前十大重仓 + 业绩比较基准对剩余仓位补全。

`
coverage_active = min(1.0, W_top10 + (1 - W_top10) * (W_benchmark_assigned + W_cash))
`

含义：
- 前十大重仓股按精确权重计入。
- 剩余仓位（1 − W_top10）按业绩比较基准结构进行补全；基准中已成功映射到行情源的部分计入覆盖率。
- 任何映射失败的基准组件（`index_code` 找不到行情源）不计入覆盖率，并把该分钟 `source_quality` 标 `degraded`。

适用条件：`lof_meta.type in ('industry', 'active')`。

### M2-B 实现约束

- 后端 M2.1 必须在 IOPV 引擎里硬编码上述两套公式，按 `lof_meta.type` 分支，不在前端二次判断。
- 前端 M5.2 仅消费 `coverage` 字段，不重新计算。
- §8 颜色阈值（≥90% 绿 / 70~90% 黄 / <70% 红）保持不变。

## §M2-B 监控告警、刷新与历史沉淀（M2 小节增补）

> 本节为 M1 联调后进入 M2 观察告警阶段的产品口径补充；不新增或修改 PRD §6 API 字段契约。

### M2-B.1 告警规则

- 告警阈值沿用当前 Settings 口径：
  - 高溢价：`premium >= +0.05`（即 +5%）
  - 高折价：`premium <= -0.02`（即 -2%）
- 阈值配置来源：前端 Settings 页面当前阈值；若用户未修改，使用上述默认值。
- 告警仅用于二级市场观察信号，不表示可直接交易，不新增申赎套利路径。
- 同一基金冷却规则沿用既有 M3：同基金 30 分钟内不重复推送。

### M2-B.2 Dashboard 展示规则

| 状态 | 判定 | 展示 |
| --- | --- | --- |
| 高溢价 | `premium >= +5%` | 红色 / 高亮信号，文案建议：`高溢价` |
| 高折价 | `premium <= -2%` | 绿色 / 高亮信号，文案建议：`高折价` |
| 中性 | `-2% < premium < +5%` | 默认样式，文案建议：`正常` |
| 数据降级 | `source_quality = degraded` | 明确提示：`数据降级，不可盲用` |
| 数据陈旧 | `source_quality = stale` | 明确提示：`数据滞后，不可盲用` |

- 当 `source_quality = degraded/stale` 与高溢价/高折价同时出现时，优先展示价格方向颜色，但必须叠加风险提示，不允许只显示交易信号。
- `active_low_liquidity` 标的在 Dashboard 可叠加低流动性提示：`低流动性，成交额不足，谨慎参考`。

### M2-B.3 刷新策略

- 非交易时段（非 09:30–11:30 / 13:00–15:00）顶部提示：`非交易时段，数据可能滞后`。
- 手动刷新不改变告警冷却规则，也不要求触发线上高频联调。

### M2-B.4 历史沉淀口径

- 本节不要求新增 §6 字段；若后续需要新增 history 粒度字段或分页字段，必须走 CCR。

### M3.3 数据源口径

| 数据类别 | 主源 | 备源/补充 | 产出字段映射 |
| --- | --- | --- | --- |
| LOF 场内价格 | 东方财富 `push2` | 新浪行情 | `price`、`ts` |
| IOPV/估算净值 | 天天基金 `fundgz` / 东方财富估值页 | 官方 NAV 仅用于收盘校准，不替代盘中估算 | `iopv` |
| 指数行情 | 免费指数行情源优先 | 东方财富/新浪可用源择优 | `coverage` 计算辅助，不新增字段 |
| 重仓行情 | 免费 A 股行情源优先 | 缺失时按基准补全降级 | `coverage`、`coverage_breakdown` |
| 源状态 | 采集器内部判定 | 本地日志与快照自证 | `source_quality` |

- `premium` 计算口径保持既有契约：`premium = price / iopv - 1`；当 `iopv` 缺失或非正数时不得产出误导性实时溢价。
- `coverage` 继续沿用 §M2 分流公式：指数型按基准指数覆盖，行业/主动型按前十大 + 基准补全 + 现金覆盖。
- `coverage_breakdown` 继续使用既有三段式结构，不新增维度字段；无法拆分时降级展示并保留已有字段形态。

### M3.4 复用 §6 字段约定

| §6 既有字段 | M3 含义 | 约束 |
| --- | --- | --- |
| `price` | LOF 场内实时/最近成交价格 | 来源为东方财富 `push2` 或新浪备源 |
| `iopv` | 盘中估算净值 / IOPV | 不用官方收盘 NAV 冒充盘中值 |
| `premium` | 实时折溢价 | 由 `price / iopv - 1` 计算 |
| `ts` | 行情快照时间戳 | 必须能反映数据时间点，不得用请求时间掩盖陈旧源 |
| `source_quality` | 当前标的数据质量 | 仅允许 `ok` / `degraded` / `stale` |
| `coverage` | 估算覆盖率 | 沿用 §M2 公式，不由前端重算 |
| `coverage_breakdown` | 覆盖率三段式明细 | 沿用前十大权重 / 基准补全权重 / 现金 |

- M3 如需内部调试字段，只能写入本地日志或本地 debug 文件，不得进入 §6 API 响应。

### M3.5 降级与失败处理

| 质量状态 | 判定口径 | 产品展示 | 数据处理 |
| --- | --- | --- | --- |
| `ok` | 主源或备源字段完整，`price`、`iopv`、`ts` 可用于计算 | 正常展示实时折溢价 | 写入本地分钟快照 |
| `degraded` | **仅看数据可得性**（PRD 1.2.1）：估算净值取不到、走 T-1 净值兜底，或行情延迟 / 主源失败转备源 / 重仓行情不全但可基准补全 | 展示“数据降级，不可盲用” | 可写入快照，并保留降级状态 |
| `stale` | **仅看数据可得性**（PRD 1.2.1）：连续两分钟取不到实时价/估值（数据源真失效）、数据时间明显滞后 | 展示“数据滞后，不可盲用” | 不得污染有效历史；可写本地日志留证 |

- 单源失败需尝试备源；单标的失败不得阻断其他标的采集。
- 三次仍失败时仅记录失败日志和 `stale` 状态，不得用上一分钟数据伪装为新鲜数据。
- 非交易时段沿用 M2 文案：`非交易时段，数据可能滞后`。
- **（PRD 1.2.1 修订）盘中 `source_quality` 不再因 `premium_error` / IOPV 偏离 T-1 净值的大小触发降级**：IOPV 与昨日净值之差是当天大盘正常波动，属有用信息，不是误差。估算准确度改由 AC-P3 盘后回测衡量。

- 不放行 AC-S1；AC-S1 仍为 hard pending，等待 3 个真实交易日配额证据。
- 不改 PRD §6 API 字段契约；字段变更必须先走 CCR。

### M3.9 历史真实沉淀（真实日线 + 30 日分位）

> 本小节为 M3.x 历史真实沉淀产品口径。目标：把详情页 `api-lof-history`（day 粒度）从 `buildFallbackHistory` 公式合成的假日线（固定 baseDate + 线性 offset）升级为**真实历史沉淀**，并以真实数据计算 `premium_close` 与 `premium_pctile_30d`。本节不改变 §6.3 返回结构（`date` / `close_price` / `official_nav` / `premium_close` / `premium_pctile_30d`）与接口名，**不触发 CCR**。

#### M3.9.1 数据源（已盘中验证可行的免费源）

| 数据类别 | 主源 | 取值 | 映射字段 |
| --- | --- | --- | --- |
| 历史场内收盘价 | 东方财富日线 K 线 `push2his`（`klt=101`，复权口径沿用既有约定） | 历史每交易日收盘价 | `close_price` |
| 历史官方净值 | 天天基金历史净值 LSJZ（`api.fund.eastmoney.com/f10/lsjz`） | 历史每披露日单位净值 | `official_nav` |

- 两源按**交易日日期对齐**后计算：`premium_close = close_price / official_nav - 1`。
- 仅使用免费源；不打线上 `next.bspapp.com`，不消耗 uniCloud RU/WU，不提交 token。
- 单只标的某一数据源不可用时，仅降级该标的对应日期，不阻断其他标的回填。

#### M3.9.2 回填范围与日常增量

- 首次回填：每只标的回填 **≥ 30 个交易日**（建议留足分位窗口，可回填更长以平滑首日分位）。
- 日常增量：daemon 在收盘后（官方净值 T+1 公布后）追加当日记录，衔接已沉淀历史，不重复回填、不覆盖已确认的历史值。
- 当日分钟快照沿用 M3.6 本地优先口径，不并入 day 粒度历史序列。

#### M3.9.3 分位口径（premium_pctile_30d）

- 定义：以**最近 30 个交易日**的 `premium_close` 为样本，计算「当前 `premium_close` 在该样本中的分位」，取值落在 `[0,1]`。
- 样本不足 30 个交易日时：
  - 标注「样本不足」（仅作前端/内部展示提示，不进入 §6.3 返回结构），分位**不展示**或返回 `null`（前端按「无值则隐藏」处理）。
  - 不得用 fallback 合成数据凑满 30 日窗口。
- 分位随历史窗口滚动更新，应满足单调性合理：`premium_close` 越高、分位越接近 1。

#### M3.9.4 数据对齐与缺日规则

- **净值 T+1 滞后**：当日已有场内收盘价、官方净值尚未公布时，该交易日 `official_nav` 留空（`null`），`premium_close` 与分位暂不计算该日，待净值公布后回补。
- **停牌 / 无成交**：当日无有效场内收盘价时，跳过该交易日，不插值、不前向填充伪造收盘价。
- **节假日**：非交易日不产生记录（与 AC-H1「节假日除外」一致）。
- **日期错配**：仅在 `close_price` 与 `official_nav` 同属一个交易日时才计算 `premium_close`；任一侧缺失则该日 `premium_close` 与分位为空。
- 缺日导致的样本减少计入 M3.9.3 的「样本不足」判定。

#### M3.9.5 存储

- 本地优先：真实历史以本地文件 / JSONL 沉淀（沿用 M3 本地链路），不上云、不耗 RU/WU。
- 线上 `lof_history` 表后置：等待 AC-S1 三个真实交易日配额证据后再由 dev-001 裁定放行，不在本阶段写线上库。

#### M3.9.6 明确不做

- 不放行 AC-S1（仍为 hard pending）。
- 不打线上 uniCloud、不做 QDII。
- 不改 §6.3 history 返回结构与接口名；不新增/删除/改名 §6 字段。
- 不用 `buildFallbackHistory` 合成数据进入真实历史序列（fallback 仅可作为「真实数据缺失时的占位降级」，必须可被识别为非真实，且不得污染分位计算）。

## 4. 页面流程与交互

### 4.1 页面清单
| 页面 | uniapp 路径 | 说明 |
| --- | --- | --- |
| Dashboard | `frontend/pages/index/index` | 30 只 LOF 实时排行 + 排序/筛选 |

### 4.2 主流程
1. 用户进入 Dashboard → 调 `api-lof-list` 拉取 30 只 LOF 当前快照。
2. 默认按"溢价率降序"排列，前端每 60 秒轮询刷新一次。
3. 点击行 → 跳详情页，调 `api-lof-detail?code=xxx`，再调 `api-lof-history?code=xxx&days=30`。
4. 详情页顶部固定显示"估算覆盖率"标签（绿/黄/红），点击展开三段式明细（前十大权重 / 基准补全权重 / 现金权重）。
5. 设置页修改阈值后写本地存储 + 通知拉取器配置文件（一期落库为可选）。

### 4.3 关键交互细节
- 列表行字段：代码、名称、现价、IOPV、溢价率、30 天分位、覆盖率。
- 溢价率正值红色、负值绿色，触阈值行加角标。
- 详情图：上半折线（现价 vs IOPV）、下半柱状（溢价率），右侧分位刻度。
- 弱网/超时：前端展示"上次刷新 mm:ss 前"+ 重试按钮。

### 4.4 详情页布局对齐竞品（PRD 1.2 产品稿）

> 本节为产品布局口径，不是前端代码；前端按本节对齐竞品（如集思录）调整详情页结构。布局变动不改 §6 字段契约。

- 顶部双列卡片：
  - 左列「场内」：现价 `price` / 涨跌幅 `price_change_pct` / 成交额 `volume_amount` / 场内份额 `circulating_shares`。
  - 右列「估值」：披露净值 `nav_official`（含 `nav_official_date`）/ 估算净值 `iopv` / **盘后估算准确度** `premium_error` 与 `nav_estimate_error_pct`（PRD 1.2.1：展示上一交易日/近 N 日的历史准确度参考，非盘中实时误差，盘中波动不计入）/ 估算溢价 `premium` / 净值溢价 `premium_nav`（净值溢价保持不变，对标集思录「净值溢价(昨收)」）。
- 中部状态标签栏：
  - 申购状态 `subscribe_status` / 赎回状态 `redeem_status`：未知或无源时不渲染该标签。
  - 申购额度/限额：`subscribe_status = limited` 且 `subscribe_limit_amount` 非 `null` 时，展示「限大额 单日 ≤ {金额}」（周期取 `subscribe_limit_period`）；`limited` 但额度为 `null` 时仅显示「限大额」标签不显示数字；开放申购/无源时不渲染该位。详情页为主，Dashboard 列表不挤入额度。
  - 低流动性：`status = active_low_liquidity` 时显示「低流动性」。
  - 数据降级：`source_quality = degraded/stale` 时显示「数据降级，不可盲用」/「数据滞后，不可盲用」。
- 下部内容区：
  - 重仓表 `holdings_top10[]`：列出 `stock_name` / `weight` / `price_change_pct` / `contribution_pct`；缺值的列项隐藏，不报错。
  - 估值时间：展示 `realtime.ts` 与 `nav_official_date`，明确区分盘中估算与披露净值。
  - 贡献度合计：可基于 `holdings_top10[].contribution_pct` 求和参考。
- 历史图保留：详情页底部保留 30 天折溢价/分位曲线（`api-lof-history`），口径同 §6.3，不变更。
- 任何字段为 `null` / 空字符串时，前端隐藏该项；不得因新字段缺失阻塞页面渲染或破坏现有 `mock-contract` / `real-api-config` 校验。

- **（M3.x 历史真实沉淀补充说明）** 返回结构、字段名与接口名保持不变；本补充仅约束数据来源与计算口径：
  - `close_price`：来自东方财富日线 K 线（真实历史场内收盘价），不再由 `buildFallbackHistory` 合成。
  - `official_nav`：来自天天基金历史净值 LSJZ（真实历史官方净值）；T+1 未公布或停牌缺日时该日可为 `null`。

- **（PRD 1.2.3 CCR：新增两字段 — 逐日累积、历史不可回填）** 在不改接口名、不动 list/detail 的前提下，§6.3 history 每条 `items[]` 新增两个**选填**字段（均可为 `null`）：
  | 字段 | 类型 | 必填 | 说明 |
  | --- | --- | --- | --- |
  | `premium_estimate_close` | number | N | 当日收盘预估溢价 = 收盘时点 `实时价 / 估算净值(IOPV) - 1`；数据源=收盘时点估算净值，禁用 `buildFallbackHistory` 合成；无收盘估值则 `null` |
  | `premium_deviation` | number | N | 溢价偏差 = `premium_estimate_close - premium_close`，即当天估算净值相对官方净值的偏差；**本质等同 PRD 1.2.1 定义的「盘后估算准确度」**（口径一致，仅展示载体不同，避免概念重复）；仅当 `premium_estimate_close` 与 `premium_close` 同日均非 `null` 时计算，否则 `null` |

  - **「前期列表大面积 `null`、随时间逐日变满」是预期行为，不是 bug**，前端按「无值则隐藏/占位」处理。
  - 缺值/时序：T 日当天 `premium_close` 尚为 `null`（净值未披露）→ `premium_deviation` 当天也为 `null`，待 T+1 回补；`official_nav` 缺失或估算净值缺失 → 对应字段 `null`。沿用 §M3.9.4 缺日规则。
  - 与 `premium_error` / `nav_estimate_error_pct` 关系：后者为 detail 层的盘后准确度展示口径；`premium_deviation` 为 history 列表的逐日同义指标，二者口径一致，不重复定义新概念。

## 9. 验收标准（可执行颗粒度，交付测试 dev-005）

> 每条均给出"测试方法 + 通过条件"，可在测试 Agent 处直接执行。

### 9.1 精度类
- AC-P1：随机抽取 5 只指数型 LOF（含 161725 / 160706 / 501050 / 160119 / 160222），盘后用官方公告净值反推真实溢价；当日每分钟样本 `|实时溢价 − 真实溢价| ≤ 0.5%`。
- AC-P2：30 只自选池整体覆盖率均值 ≥ 90%；个体 `< 70%` 数量 ≤ 3。其中指数型 LOF 覆盖率应 ≈ 1.00（绿）。
- AC-P3（PRD 1.2.1 修订）：**盘后估算准确度校验**（历史置信度指标，不参与盘中实时降级）。
  - 概念澄清：盘中 `iopv` 与 `nav_official`（T-1 昨日净值）之差主要由当天大盘涨跌造成，是**正常盘中波动**，是有用信息，**不得据此降级**；估算准确度只能用当日真实披露净值（T+1 才公布）回测。
  - 校验时点：收盘后，待当日官方净值（T+1）公布。
  - 公式：`估算准确度误差 = |当日收盘估算净值 - 当日官方净值| / 当日官方净值`。
  - 阈值：盘后误差 `≥ 1%` 标记该基金「估算偏差较大」，作为**置信度提示**展示，**不**触发实时 `source_quality` 降级。
  - 测试方法：取某交易日收盘估算净值与该日 T+1 公布的官方净值，逐只回算；通过条件：抽样 ≥10 只，误差计算正确，`≥1%` 的标的被标记「估算偏差较大」且未触发盘中 `stale`。
  - 反例守门：在盘中大盘单边波动场景下，AC-P3 不得把任何「数据源正常但 IOPV 偏离 T-1 净值」的标的判为 `degraded/stale`（对应 6/22 真盘 degraded=13/stale=90 误报问题，修订后应为 0 误报）。
- AC-P4：净值溢价公式校验。`premium_nav` 必须等于 `(price - nav_official) / nav_official`，与离线脚本计算的双精度结果误差 `≤ 1e-4`。测试方法：按 `lof-list` 抽 ≥30 行，逐行重算并对比；通过条件：100% 行误差 `≤ 1e-4`。
- AC-P5：字段兜底渲染。新字段（`subscribe_status` / `redeem_status` / `fund_scale` / `circulating_shares` / `holdings_top10[].price_change_pct` 等）为 `null` 或空字符串时，前端不渲染该项且不报错；旧字段（如 `coverage_top10` / `coverage_breakdown` / `benchmark_raw`）必须照常渲染。测试方法：mock 一组将上述新字段全部置 `null` 的样例；通过条件：详情页/列表正常渲染，控制台无报错，`mock-contract` 校验通过。

- AC-P6（PRD 1.3）：申赎状态枚举受限 + 主备源/兜底。`subscribe_status` ∈ {`open`,`limited`,`suspended`,`closed`,`unknown`}、`redeem_status` ∈ {`open`,`suspended`,`closed`,`unknown`}；主源取 `fundmob_basic`（SGZT/SHZT），主源失败回退概况页 HTML，仍失败兜底 `unknown`；测试方法：抽 ≥5 只标的校验返回值落在枚举集内，且「开放申购→open / 限大额→limited / 开放赎回→open」映射正确；通过条件：无越界枚举、映射正确、断源时为 `unknown` 不报错。
- AC-P7（PRD 1.3）：申购额度独立解析。`subscribe_limit_amount` 仅当申购状态为 `limited` 且原始文案含具体数字时解析为金额（元）+ `subscribe_limit_period=day`；开放申购/无数字的限大额/无源时为 `null`；**禁止把哨兵值 `MAXSG=1000亿` 当真实额度**（须为 `null`）；测试方法：用探测样本（161725=50万、161005=2万、160632=20万有数字；160706/501050/501090 开放=null；501203 限大额无数字=null）逐只校验；通过条件：有数字标的解析金额+周期正确，其余全 `null`，状态与额度互不推断。

- AC-H1：每只 LOF 详情页可绘制最近 30 天溢价折线，无断点（节假日除外）。
- AC-H2：`premium_pctile_30d` 数值落在 `[0,1]`，与离线脚本计算结果误差 `≤ 0.01`。

- AC-H3（M3.x）：真实历史沉淀。`api-lof-history`（day 粒度）返回的 `close_price` / `official_nav` 来自真实免费源（东财日线 / 天天基金 LSJZ），不再由 `buildFallbackHistory` 合成；测试方法：抽 ≥3 只标的，校验序列不含固定 baseDate 线性 offset 特征，且与源数据一致。
- AC-H4（M3.x）：`premium_close` 计算正确性。抽样 ≥30 个 (date, close_price, official_nav) 记录，逐条重算 `close_price / official_nav - 1`，与返回值误差 `≤ 1e-4`；缺 `official_nav` 的交易日 `premium_close` 应为 `null`，不参与误差比对。
- AC-H5（M3.x）：分位样本不足处理。真实样本 `< 30` 个交易日时，`premium_pctile_30d` 不得由合成数据凑满，应标注「样本不足」并允许 `null`/不展示；样本 `≥ 30` 时分位落在 `[0,1]` 且与离线脚本误差 `≤ 0.01`，单调性合理。
- AC-H6（PRD 1.2.3）：`premium_estimate_close` 来源校验。其值=收盘时点估算净值算出的 `实时价 / IOPV - 1`，禁用 `buildFallbackHistory` 合成；测试方法：抽 ≥3 只标的近 N 个交易日记录，校验该字段在启用日之前为 `null`、启用日起逐日出现且非合成；通过条件：无合成特征，启用前历史均为 `null`。
- AC-H7（PRD 1.2.3）：`premium_deviation` 公式与时序。`premium_deviation == premium_estimate_close - premium_close`，与离线重算双精度误差 `≤ 1e-4`；仅当二者同日均非 `null` 时计算，任一为 `null`（如 T 日净值未披露）则 `premium_deviation` 必须为 `null`，不得置 0 或伪造；通过条件：抽样逐条对比，公式误差达标且缺值场景全为 `null`。

## 11. 风险与硬约束

- R1：免费数据源限速 / 字段变更 → 拉取器需指数退避 + 多源切换；元数据每日校准。
- R2：uniCloud 免费额度（云函数 5 万/日 / 库读 5 万 / 库写 3 万）→ 拉取数据放本机，uniCloud 仅做"被前端读 + 接 ingest 写"；批量写入合并 30 只为一次请求。
- R3：重仓股仅披露前十 → 详情页强制三段式覆盖率展示（AC-T1）。
- R4：watchlist v2 已固化，但含 `active_low_liquidity` 低流动性候选；后续如全市场扫描或真实成交复核发现更优标的，由 dev-002 在 `watchlist-v3` 中整列替换。
- R5：一期不做申赎套利 / 自动下单 / 多用户。
- R6（PRD 1.2 → 1.3 更新）：`subscribe_status` / `redeem_status` 已由 dev-004 探测确认稳定免费源（天天基金移动端 `fundmob_basic`，7/7 全覆盖），PRD 1.3 转正为真实值（主源 + 备源 HTML + 兜底 `unknown`）；`subscribe_limit_amount`/`subscribe_limit_period` 部分可解析（限大额带数字时），无源/开放/哨兵值时 `null`。`fund_scale` / `circulating_shares` 仍无稳定分钟级免费源，维持 `null` 并以 `scale_yi` 兜底；如后续找到稳定源再走 CCR 升 PRD 1.4。
- R7（PRD 1.2.1）：误报风险。盘中以净值溢价/估算误差为由的降级会造成满屏「数据滞后」误报（6/22 实证）。硬约束：盘中降级仅依据数据可得性；`premium_error`/`nav_estimate_error_pct` 仅展示盘后准确度，dev-004 降级逻辑与 dev-005 AC-P3 验收须按本口径对齐。
- R8（PRD 1.2.3）：历史预估溢价不可回填、逐日累积。免费源无历史估算净值序列，`premium_estimate_close` / `premium_deviation` 启用日之前的历史只能为 `null`，列表会出现「前期大面积 null、随时间逐日变满」——这是**预期行为，不是 bug**；前端须按「无值则隐藏/占位」展示并避免误判为缺陷，dev-004 沉淀逻辑须只追加当日、不伪造历史。

#### 12.7.1 写权所有制（Single-Writer Map）

每个文件 / 目录有且仅有一个"写权所有者"，其他人只读不写；要改必须发"契约变更请求 (CCR)"或"建议 PR"给所有者。

#### 12.7.2 接口契约冻结（Contract Lock）

- §6 API 契约一旦 PRD 定档即 **冻结**。任何字段增删改必须走 **CCR（Contract Change Request）**：
  1. 提出方向 dev-001 发跨会话消息：`【CCR-XX】请求变更 <接口名>.<字段>`，含「现状 / 提议 / 影响范围 / 紧急度」。
  2. dev-001 召开三方对齐（前端 / 后端 / PM）。
  3. 通过则 dev-002 升 PRD（§6 + 变更记录），dev-001 squash merge 并打里程碑 tag（如 `prd-v1.2`）。
  4. dev-001 跨会话广播给前端 / 后端 / 测试，要求 24h 内 rebase。
- 未走 CCR 自创字段者，提交将被项目经理驳回。

#### 12.7.3 Mock 优先并行开发（Mock-First Parallel）

- 前端 dev-003 在后端接口未上线前，必须基于 PRD §6 字段写 **完整 mock**，字段名与契约 100% 一致。
- 后端 dev-004 第一周必须先出 cloudfunctions 空骨架（`return mock`），让前端能联调。
- 后端真实实现完成后，前端 **只删 mock 不改业务代码**。

#### 12.7.4 主干快进（Trunk Sync ≤ 24h）

- 每个 `feat/*` 分支必须 **每天上班第一件事 rebase main**：

```bash
git checkout feat/your-branch
git fetch origin
git rebase origin/main           # 不用 merge，保线性历史
git push --force-with-lease      # 安全 force push
```

- 分叉 main 超过 24h 的分支，dev-001 每日 18:00 巡检后会发提醒；超过 72h 强制要求拆分。

#### 12.7.5 串行合并（Serialized Merge）

- 项目经理 dev-001 是 **唯一合并人**，merge 操作 **串行不并行**。
- 一次只 squash merge 一个分支 → push main → 广播 → 等所有持有 `feat/*` 分支的 Agent 完成 rebase → 才 merge 下一个。
- 大模块（如 `cloud-api-skeleton`）合并前必须有 dev-005 的 Smoke Test 报告。

#### 12.7.6 跨会话广播（Broadcast on Merge）

- 每次 squash merge 完成后，dev-001 必须跨会话广播：

```
【From: 项目经理(dev-001)】【To: 全员】
🔔 main 已更新到 <commit>: <说明>
影响：
- 接口字段变更：…（无 / 列出）
- 公共依赖变更：…（无 / 列出）
请所有持有 feat/* 分支的 Agent 在 24h 内 rebase main。
```

#### 12.7.7 派工指令是单一信号源（Single Source of Tasking）

- **任何 Agent 在收到 dev-001 正式开工指令前，不得创建 `feat/*` 分支或撰写功能代码**。
- 正式指令必须含：阅读范围 / 工作目录 / 技术栈硬约束 / 交付清单 / 分支命名 / 验收 AC 编号 / 时间预期。
- 草稿、讨论、口头承诺均不构成开工依据。
- 违反者由 dev-001 走流程裁决：未提交内容 stash 暂存，已提交内容须补"自检报告"承认违规并冻结合并。
