# 01-需求 / PRD 1.0（LOF 实时溢价监控）

> 版本：PRD 1.0｜状态：定档｜责任人：产品经理(dev-002)｜锁定日期：2026-06-18

## 快速摘要

- 背景：场内 LOF 现有炒股软件展示的溢价基于 T-1 净值，盘中失真 1~2%，吃掉全部可交易信号。
- 核心目标：自建"档 3 + 业绩比较基准补全（工业级）"实时溢价计算与监控系统，目标精度 ≥ 90%。
- 一期范围：仅国内 LOF（QDII 二期）／30 只热门自选池／1 分钟刷新／网页 Dashboard + 阈值推送／30 天分位回看／仅二级市场溢价路径。
- 架构：本机 Python 拉取器 + uniCloud 免费云空间 + uniapp 前端（H5 → 小程序）。
- 配套交付：`assets/lof-watchlist-v1.csv`（30 只候选池）、`assets/benchmark-mapping-v1.csv`（基准结构化映射）。

## 1. 背景与目标

### 1.1 痛点
- 软件溢价 = (现价 − 昨日净值) / 昨日净值，T-1 数据导致盘中失真。
- LOF 可交易套利空间多在 1~3%，与误差同量级，信号几乎不可用。

### 1.2 目标
1. 盘中每分钟输出 30 只 LOF 真实实时溢价率，整体覆盖率 ≥ 90%。
2. 提供 30 天折溢价历史分位，支持判断当前溢价相对位置。
3. 阈值触发时主动推送告警，避免人工盯盘。
4. 前端使用 uniapp 框架，一期 H5，二期可平滑迁移到微信小程序。

## 2. 一期范围与边界

### 2.1 本期包含
- 标的：国内 LOF（普通股票/债券/混合型 LOF），30 只自选池。
- 算法：档 3 + 业绩比较基准补全。前十大重仓股按季报权重 × 实时股价；剩余仓位按业绩比较基准指数实时同步。
- 刷新：盘中（09:30–11:30 / 13:00–15:00）每 1 分钟；盘后用官方净值校准。
- 数据源：东方财富 / 新浪 / 天天基金 / 集思录 公开免费接口。
- 历史：30 天折溢价分布 + 当前分位百分比。
- 套利路径：仅二级市场买卖溢价。
- 告警：溢价 > 5% 或折价 < −2%；同基金 30 分钟冷却；推送 Server酱或邮件二选一。
- 前端：uniapp H5。

### 2.2 本期不包含
- QDII LOF、申赎套利、全市场扫描、付费数据源、自动下单、多用户体系。


## 3. 功能模块结构

`
LOF 实时溢价监控（一期）
├─ M1 数据采集（拉取器）
│  ├─ M1.1 行情拉取：LOF 现价 / 重仓股现价 / 指数现价
│  ├─ M1.2 元数据拉取：重仓股权重、业绩比较基准、规模、官方净值
│  ├─ M1.3 调度：apscheduler 盘中每分钟，盘后 18:00 校准
│  └─ M1.4 写入：通过 ingest 接口写 uniCloud
├─ M2 实时估值与溢价计算
│  ├─ M2.1 IOPV 估算（档 3 + 基准补全）
│  ├─ M2.2 溢价率计算 = (现价 − IOPV) / IOPV
│  ├─ M2.3 估算覆盖率 = 前十重仓权重之和（透明展示）
│  └─ M2.4 历史溢价 30 天分位（每日校准时刷新）
├─ M3 告警引擎
│  ├─ M3.1 阈值判断（溢价/折价默认 +5% / −2%）
│  ├─ M3.2 冷却控制（同基金 30 分钟）
│  ├─ M3.3 推送通道（Server酱 / 邮件，可配）
│  └─ M3.4 告警日志（alert_log）
├─ M4 后端 API（uniCloud 云函数）
│  ├─ M4.1 list   列表 + 实时溢价
│  ├─ M4.2 detail 详情 + 30 天分位 + 估算覆盖率
│  ├─ M4.3 history 历史 K 线
│  └─ M4.4 ingest  拉取器写入（签名）
└─ M5 前端（uniapp H5）
   ├─ M5.1 Dashboard：30 只 LOF 实时排行（按溢价率降序）
   ├─ M5.2 详情页：曲线 + 分位 + 估算覆盖率标签 + 重仓股
   └─ M5.3 设置页：阈值 / 推送通道（写本地存储 + ingest 同步）
`

## 4. 页面流程与交互

### 4.1 页面清单
| 页面 | 路径 | 说明 |
| --- | --- | --- |
| Dashboard | `/pages/index` | 30 只 LOF 实时排行 + 排序/筛选 |
| 详情页 | `/pages/detail/[code]` | 单只 LOF 实时溢价 / 30 天曲线 / 分位 / 重仓股 / 覆盖率 |
| 设置页 | `/pages/settings` | 阈值、推送通道、自选池开关 |

### 4.2 主流程
1. 用户进入 Dashboard → 自动调用 `api/lof/list` 拉取 30 只 LOF 当前快照。
2. 默认按"溢价率降序"排列，前端每 60 秒轮询刷新一次。
3. 点击行 → 跳详情页，调 `api/lof/detail?code=xxx` 拉详情，再调 `api/lof/history?code=xxx&days=30` 拉历史。
4. 详情页顶部固定显示 `估算覆盖率` 标签（绿/黄/红：≥90% / 70~90% / <70%）。
5. 设置页修改阈值后，POST 到 `api/lof/settings`（一期可仅写本地存储 + 通知拉取器配置文件，落库为可选）。

### 4.3 关键交互细节
- 列表行字段：代码、名称、现价、IOPV、溢价率、30 天分位、覆盖率。
- 溢价率正值红色、负值绿色，触阈值行加角标。
- 详情图：上半折线（现价 vs IOPV）、下半柱状（溢价率），右侧分位刻度。
- 弱网/超时：前端展示"上次刷新 mm:ss 前"+ 重试按钮。

- 用途：拉取器写入（签名）
- Header：`X-Ingest-Token: <UNICLOUD_INGEST_TOKEN>`
- 请求体：
  `json
  {
    "ts": "2026-06-18T10:31:00+08:00",
    "items": [
      {

### 6.5 通用错误码
| code | 含义 |
| --- | --- |
| 0 | 成功 |
| 4001 | 参数缺失或非法 |
| 4010 | 鉴权失败 |
| 4040 | 资源不存在（如 code 不在自选池） |
| 4290 | 写入限流（uniCloud 配额接近上限） |
| 5000 | 服务异常 |

### 7.2 业绩比较基准映射
- 文件：`assets/benchmark-mapping-v1.csv`
- 字段：`code, name, component_name, index_code, weight`
- 说明：每只 LOF 按"组成项"展开为多行，权重之和应 = 1.0；现金/活期统一标记 `index_code = CASH`。
- 校验规则：
  1. `sum(weight) per code == 1.00 ± 0.001`
  2. `index_code` 必须能在拉取器索引清单中找到对应行情源；找不到时降级为"使用最大权重指数全替"，并把 `source_quality` 标 `degraded`。

## 8. 估算覆盖率与降级策略

| 覆盖率 | UI 标签颜色 | 含义 | 降级 |
| --- | --- | --- | --- |
| ≥ 90% | 绿 | 工业级，达标 | 无 |
| 70%~90% | 黄 | 可参考但非高置信 | 前端弹"覆盖率偏低"提示 |
| < 70% | 红 | 不可用作交易信号 | 详情页禁用"加入告警" |

- 当某只 LOF 重仓股或基准指数行情拉取失败时，`source_quality = degraded`；连续两分钟失败标 `stale` 并跳过本分钟写入。

## 9. 验收标准（可执行颗粒度，交付测试 dev-005）

> 每条均给出"测试方法 + 通过条件"，可在测试 Agent 处直接执行。

### 9.1 精度类
- AC-P1：随机抽取 5 只指数型 LOF（含 161725 / 160706 / 501050 / 160119 / 160222），盘后用官方公告净值反推真实溢价；当日每分钟样本 |实时溢价 − 真实溢价| ≤ 0.5%。
- AC-P2：30 只自选池整体覆盖率均值 ≥ 90%；个体 < 70% 数量 ≤ 3。

### 9.2 完整度类
- AC-C1：盘中 09:30–11:30 / 13:00–15:00 每分钟 `lof_realtime` 应有 30 行数据；允许 ≤ 3 次／日整分钟缺失。
- AC-C2：拉取器单次任务失败可在 30 秒内完成 3 次重试；3 次仍失败写日志并跳过本分钟。

### 9.3 历史与分位
- AC-H1：每只 LOF 详情页可绘制最近 30 天溢价折线，无断点（节假日除外）。
- AC-H2：`premium_pctile_30d` 数值落在 [0,1]，与离线脚本计算结果误差 ≤ 0.01。

### 9.4 告警
- AC-A1：模拟某只 LOF 溢价 > 5% 持续 1 分钟，可在 ≤ 2 分钟内通过配置通道收到一条告警。
- AC-A2：30 分钟内对同一基金多次触发，仅推送 1 次；alert_log 应有冷却拦截记录。
- AC-A3：盘中（非交易时段不推）；非交易时段触发应被忽略。

### 9.5 接口
- AC-I1：`api/lof/list` p95 响应 ≤ 800ms；返回 30 行结构合规。
- AC-I2：`api/lof/detail` 包含 `coverage_top10` / `benchmark_components` / `holdings_top10` / `realtime` / `pctile_30d` 5 个块。
- AC-I3：`api/lof/history` 在 `days=30` 下返回不少于 20 个交易日。
- AC-I4：`ingest/realtime` 在 token 错误时返回 4010；字段缺失返回 4001；正常写入返回 `accepted == 提交条数`。

### 9.6 架构与配额
- AC-S1：uniCloud 云函数日调用 ≤ 50000、库读 ≤ 50000、库写 ≤ 30000；连续 3 个交易日真实统计达标。
- AC-S2：拉取器写入仅走 ingest 接口；前端不直连拉取器；HBuilderX 可一键编译 H5。
- AC-S3：前端 uniapp 工程在不改源码的前提下，可通过 `npm run build:mp-weixin` 完成微信小程序构建（不要求上线）。

### 9.7 透明度（硬约束落实）
- AC-T1：详情页头部固定显示"估算覆盖率"标签（数值 + 颜色），并在 hover/tap 时弹出"前十重仓权重和 + 基准补全比例"明细。

## 10. 关键决策记录

| 决策项 | 选择 | 理由 |
| --- | --- | --- |
| 标的范围 | 仅国内 LOF | 聚焦核心问题，控制一期范围 |
| 算法精度 | 档 3 + 业绩比较基准补全 | 工业级 90~95% 精度，工程量可控 |
| 刷新频率 | 1 分钟 | 兼顾时效与免费数据源限速 |
| 套利路径 | 仅二级市场溢价 | 一期不做申赎，规避费率/到账复杂度 |
| 推送通道 | Server酱 / 邮件二选一 | 个人用户最低成本 |
| 架构 | 本机 Python + uniCloud + uniapp | 免费额度 + H5 平滑迁小程序 |

## 11. 风险与硬约束

- R1：免费数据源限速 / 字段变更 → 拉取器需指数退避 + 多源切换；元数据每日校准。
- R2：uniCloud 免费额度（云函数 5 万/日 / 库读 5 万 / 库写 3 万）→ 拉取数据放本机，uniCloud 仅做"被前端读 + 接 ingest 写"；批量写入合并 30 只为一次请求。
- R3：重仓股仅披露前十，覆盖率不足时不得隐藏 → 详情页强制展示"估算覆盖率"标签（AC-T1）。
- R4：一期不做申赎套利 / 自动下单 / 多用户。
